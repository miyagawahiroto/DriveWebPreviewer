# CASA - 一般公開のための年次セキュリティ評価（restricted スコープ）

`drive.readonly` は **restricted（制限付き）スコープ**のため、OAuth 同意画面を **「外部(External)＋本番(In production)」** で一般公開するには、Google の OAuth 検証に加えて **CASA（Cloud Application Security Assessment）= 年次の第三者セキュリティ評価**を通す必要がある。本書はその申請・通過・更新の手順をまとめる。

> **役割分担（CLAUDE.md 準拠）**: Google Cloud Console の検証提出・同意画面設定・評価機関とのやり取り・費用承認は**人間（管理者）が行う**。AI は本手順書の整備、申請素材（説明文・データフロー図のたたき台）の作成、コード側のセキュリティ要件確認までを担当する。
>
> **注意**: CASA のポータル URL・料金・認定ラボの一覧・Tier 判定は Google / App Defense Alliance 側で随時更新される。**最終的な正は、検証提出後に Google から届くメールに記載される CASA 案内（リンク・期限・要求 Tier）**とする。本書は全体像と準備事項の整理用。

---

## 0. そもそも CASA が要るのか（最終確認）

| 条件 | CASA |
|------|------|
| 外部(External)＋本番＋`drive.readonly` | **必要**（本書の対象） |
| 内部(Internal) / 管理コンソールで「信頼済み」/ テスト中(≤100) | 不要（`DISTRIBUTION.md`） |
| `drive.file` に移行 | 不要（`PICKER.md` の方式・要改修） |

→ **一般公開を続ける前提**なら CASA を通すか、`drive.file` 化で回避するかの二択。本書は前者の手順。

## 1. 全体フロー

```
[準備] 申請素材を揃える
   ↓
[ルートB-1] GCP で OAuth 検証を提出（ブランド+スコープ+デモ動画）
   ↓
Google が「restricted スコープ → CASA 必須」とメール通知（期限つき）
   ↓
[ルートB-2] CASA を実施（App Defense Alliance 経由）
   ├─ パスA: セルフスキャン（SAST 等を自分で実施しアップロード）… 安価・小規模向き
   └─ パスB: 認定ラボによる独立評価 …… 高コスト・規模が大きい/高リスク時
   ↓
OWASP ASVS L2 ベースで検査 → 指摘を是正
   ↓
Letter of Validation（LoV）発行 → Google へ連携
   ↓
Google が検証完了 → 一般公開で警告なし
   ↓
[毎年] 12 か月ごとに再評価（LoV 更新）
```

ルート A（Chrome ウェブストアの審査）は別系統。両方そろって「誰でもインストール＋ログインで利用」になる（`PUBLISHING.md`）。

## 2. 事前準備（検証提出の前に揃える）

| 項目 | 状態 | 補足 |
|------|------|------|
| 公開された**プライバシーポリシー URL** | **未整備**（`docs/PRIVACY.md` が無い） | **要作成 → Web 公開して URL 化**。CASA・OAuth 検証の必須要件 |
| **ドメイン所有権の確認** | 要対応 | プライバシーポリシー/ホームページのドメインを Search Console 等で確認 |

## 3. ルート B-1: OAuth 検証の提出（人間）

1. Google Cloud Console（プロジェクト `162512121004`）→「Google Auth Platform」→ 対象・データアクセス・ブランディングを確定
2. スコープに `https://www.googleapis.com/auth/drive.readonly` が登録されていることを確認
3. **「アプリを公開」**（テスト中 → 本番）して**検証を申請**
4. 各スコープの利用目的・デモ動画・プライバシーポリシー URL・ドメイン所有権を提出
5. Google のレビュー後、restricted スコープに対して **CASA 要求のメール**（実施期限・ポータルリンク・要求 Tier 付き）が届く

> 期限内に CASA を完了しないと、restricted スコープの新規ユーザー認可が制限される場合がある。早めに着手する。

## 4. ルート B-2: CASA の実施

### 4.1 Tier とベンチマーク
- Google の restricted スコープは通常 **Tier 2** が要求される（メール記載の指示が正）
- 検査ベンチマークは **OWASP ASVS（Application Security Verification Standard）Level 2** 相当
- 運営は **App Defense Alliance（ADA）**。Google のメールから CASA ポータルへ進む

### 4.2 パス選択
| パス | 内容 | 向き | コスト感（目安・要確認） |
|------|------|------|------------------------|
| **セルフスキャン** | 承認済み SAST ツールでコード解析し、結果をポータルにアップロード。ラボが確認 | バックエンドが無い小規模アプリ（本アプリ向き） | 低（無償〜数百ドル規模のことが多い・要確認） |
| **認定ラボの独立評価** | NCC Group / Bishop Fox / Leviathan / Tevora 等の認定ラボがペンテスト等を実施 | 規模大・高リスク | 高（数千ドル〜・要確認） |

> 本アプリは**外部サーバーを持たず通信先が Google API のみ**で攻撃面が小さいため、**セルフスキャン**で通せる可能性が高い。最終的な要求はメール指示に従う。

### 4.3 提出物
- アプリ構成図 / データフロー（本アプリ: ブラウザ内完結・外部送信なしを明示。`ARCHITECTURE.md` を基に作成）
- スコープと取得データの説明
- **SAST スキャン結果**（対象: 本リポジトリの TypeScript ソース＋ビルド成果物）
- 必要に応じて DAST / 構成レビュー

### 4.4 是正 → LoV
1. 検出された指摘を是正（多くは設定・依存パッケージ更新・ドキュメント補強）
2. 再スキャンで解消を確認
3. ラボが **Letter of Validation（LoV）** を発行し Google に連携
4. Google が OAuth 検証を完了 → 「未確認アプリ」警告が消える

## 5. 本アプリの ASVS 対応状況（強み・確認ポイント）

CASA（ASVS L2）で問われる主要項目と本アプリの実態。多くは**設計上すでに満たしている**（`SECURITY.md`）。

| ASVS 観点 | 本アプリの状態 | 確認/対応 |
|-----------|--------------|-----------|
| 秘密情報のハードコード禁止 | クライアントシークレット不使用（implicit flow）。`client_id` は公開値 | ◎ `auth.ts` |
| トークンのログ出力・外部送信禁止 | `console.log` 出力禁止・外部送信なし | 要レビュー（ログに token が出ていないか） |
| トークンの保存 | `chrome.storage.session` のみ（揮発）。永続平文保存なし | ◎ `auth.ts` |
| 通信の TLS / 宛先限定 | `https://www.googleapis.com/*` のみ | ◎ `manifest.json` host_permissions |
| 権限最小化 | `identity` / `storage` のみ、`content_scripts` は drive.google.com 限定 | ◎ `manifest.json` |
| 取得コンテンツ実行リスク | `chrome-extension://` で HTML/JS 実行（拡張ページ CSP 適用） | 留意点として説明（`SECURITY.md` 3.5/4）。サンドボックス化は将来課題 |
| 依存パッケージの既知脆弱性 | ビルド依存（esbuild 等） | 申請前に `npm audit` で確認・更新 |

> 「取得コンテンツを拡張オリジンで実行する」点は評価で問われ得る。データフロー図で「対象は信頼できる自分の Drive ファイル」「外部に出さない」ことを明示し、`SECURITY.md` の対策方針を添えると説明しやすい。

## 7. 更新（年次）

- LoV の有効期限は概ね **12 か月**。期限前に再評価（再スキャン・再申請）が必要
- スコープ・データフロー・依存に変更があれば再評価時に反映
- 更新を失念すると restricted スコープの利用が制限されうるため、期限をカレンダー管理する
