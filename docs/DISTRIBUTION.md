# DISTRIBUTION - 配布フェーズ・OAuth スコープ・Google 審査の方針

本プロダクトは **OAuth（公式 Drive API）** を採用する。OAuth なし（Cookie 相乗り等）の方式は、一般公開・保守性・規約の観点から不採用とする。

エンドユーザーの操作は全フェーズ共通で **「インストール ＋ Google ログイン（許可）」だけ**。OAuth クライアント ID は全ユーザー共通・不可視で、ユーザーは設定しない。

## 1. 配布フェーズ

| フェーズ | 対象 | 同意画面 | Google 審査 |
|---------|------|---------|-------------|
| **Phase 1（現在）** | 社内（または自分のテスト） | 「内部(Internal)」または「外部(External)＋テスト」 | **不要** |
| **Phase 2** | 不特定多数の一般 Google ユーザー | 「外部(External)＋本番(In production)」 | **必要**（スコープに依存。下記） |

- 「内部」は Google Workspace 組織内の全員が即利用可（審査不要）
- 「外部＋テスト」はテストユーザー（最大 100 名）のみ利用可（審査不要）

## 2. OAuth スコープと審査の対応（Phase 2 で重要）

| スコープ | アクセス範囲 | 分類 | 一般公開時の審査 |
|---------|------------|------|-----------------|
| `drive.readonly` | ユーザーの**全ファイル**を閲覧 | **restricted（制限付き）** | アプリ検証＋**年次のセキュリティ評価（第三者 CASA・有償になりうる）** |
| `drive.file` | ユーザーが **Picker で選んだファイル/フォルダ＋その配下**のみ | recommended（推奨） | 軽い（restricted の重い評価が不要） |

> いずれのスコープでも **Phase 1（内部/テスト）では審査不要**。差が出るのは Phase 2（外部・本番公開）。

## 3. 推奨プラン

### Phase 1（今すぐ）

- 既存実装のまま **`drive.readonly`** を使用
- OAuth 設定は `SETUP_OAUTH.md` の手順（拡張機能 ID 確認 → Drive API 有効化 → クライアント ID 発行 → manifest 反映）
- 同意画面を「内部」または「テスト」にして、審査なしで社内・自分で動作確認

### Phase 2（一般公開の直前に判断）

二択：

1. **`drive.file` ＋ Google Picker に移行**（推奨。審査が軽く、ユーザーが選んだフォルダ配下＝サブフォルダもそのまま解決できる）
2. **`drive.readonly` のまま公開審査を通す**（年次セキュリティ評価のコスト・期間を許容できる場合）

#### `drive.file` 化の検討事項（Phase 2）

- フォルダのアクセス許可は **Google Picker** でユーザーに選ばせる方式になる
- **MV3 は外部ホストのコード実行を制限**するため、Picker ライブラリ（`apis.google.com` 由来）の読み込みに一工夫が必要（サンドボックスページ等）。Phase 2 着手時に PoC で検証する
- content-script の「Drive 画面のボタン」起点から、Picker 起点の導線への変更が必要

## 4. ユーザー体験（全フェーズ不変）

1. 拡張機能をインストール
2. プレビュー操作
3. （初回のみ）Google ログイン＋許可

クライアント ID・スコープ・審査などはすべて開発者側の関心事で、ユーザーには見えない。
